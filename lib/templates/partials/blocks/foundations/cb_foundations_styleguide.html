CSS Style Guide

Coding Style

Spacing

  • Use soft-tabs with a four space indent (for readability and consistency across all of our codebases).
  • Use a single space after : in property declarations.
  • Use a single space before { in selector rule declarations.
  • Every CSS selector rule should reside on its own line to improve commit diffs and tracking changes to our CSS.
  • Use a single newline after every CSS { ... } block within a given section.
  • Use two newlines to separate a given CSS section from the next CSS section in a file. Each section should start with a comment describing the section's purpose.

Commenting

Document your styles with comments as often as possible following KSS syntax. Long-term, this KSS documentation will be used in the living styleguide so we don't have to manage the documentation separately from the code.

For Styleguide reference definitions, we use the dot notation format of Styleguide Section.Reference (i.e. for the button element base styling, the reference definition would be Styleguide Elements.Interactive.Buttons).

Use // for comment blocks (instead of /* */). Compass will automatically strip these out of production code.

Variables

All $variable names should be formatted as $categoryName-subcategoryName-variantName where subcategoryName is optional and all sections in a variable should be camelCase.

We use a number of categories of variables and mixins in our styles. Each of these should reside in their category specific files located within the /styles/base/foundations/ folder. Non-category specific $variable or @mixin values used in more than one file should be placed in the styles/base/_base.scss file. Any $variable or mixin used solely in one file should be included at the top of the relevant file.

Color

All color definitions (base and derivative) should reside in the global _colors.scss file. Derivatives of colors may be created using SASS color functions and Compass color helpers, but this should be done with caution. When reusing a color for a component-specific purpose, the $variable should be defined at the top of the file in which the component-specific color is used.

When creating a base color definition, you should use hex for opaque values and rgba for semi-transparent values.

Avoid creating new color definitions if possible, as we want to limit the number of colors used throughout the site to increase consistency and provide structure around our brand identity. Refer to the Brand documentation for specifics around color usage.

Z-Index

Our Z-Index scale is defined in _zindex.scss, and it provides a total of nine indices spread across three categories: body, overlay, and underlay. Each of these categories has a specific purpose:

  • body: Provides mid-range z-index values used for the body of any current interface. The majority of the site is set to the z-body-mid value.
  • overlay: Provides high z-index values used to create the effect of being above a given interface. These are primarily used for popups, toasts, floating headers, and similar components.
  • underlay: Provides low z-index values used to create the effect of being beneath a given interface. These are primarily used for background components.

Stacking order is a uniquely odd thing. If you are running into issues, please read "What No One Told You About Z-Index", as it's a great overview of how stacking order works. Pay particular attention to the Stacking Contexts. Ideally, we should create stacking contexts using position and z-index as opposed to using the opacity hack.

Typography

All typographical base definitions should reside in the global _typography.scss file. Most important in this section is the typography() mixin. This allows us to use any combination of our predefined typeface choices throughout the site without having to refactor changes to our typography in multiple places.

The mixin provides four variants:

  • headline: This is used to style all of our h* elements with our headline typography choices. It is acceptable but not encouraged to style non-h* elements as headlines.
  • subhead: Subheaders, typically meant to be paired with Headlines, should be a h* element.
  • accent: Accents should be used judiciously for short non-interface copy. Examples include blockquotes, pull quotes, and callouts.
  • body: Everything is prestyled as body type, so this mixin keyword is typically only used to override other typeface styles.

For more about our typography, please refer to brand typography, typography in use, and vertical rhythm.

Media Queries

We use breakpoint to define media queries. Specific breakpoints should be defined in the _mq.scss file. This allows us to fine-tune media queries across the site all at once. If a component needs a specific media query breakpoint, define it at the top of the file in which the component-specific breakpoint is used.

Formatting

Nesting

Avoid nesting deeper than three levels, as this can cause performance issues if we do it too often. If you find yourself needing nesting deeper than three levels, consider refactoring the styles into smaller components or simplifying the DOM structure.

To learn more about writing efficiently nested CSS, Mozilla has a good guide on the subject.

Quotes

While quotes are optional in CSS, we use double quotes to visually distinguish strings from variables, selectors, and properties.

Browser Prefixing

All browser prefixes for CSS3 properties should never be created manually. Instead, you should use the Bourbon Mixin to create the property with the necessary prefixes. Check out Can I Use for the latest browser support matrix on the property and ensure you actually need browser prefixes.

File Organization

Our Styles folder for any given project should always have the same structure:

  • /styles
    • /base
      • /components
        • _component.scss
      • /foundations
        • _foundation.scss
      • /plugins
        • _plugin.scss
    • /project
      • /components
        • _component.scss
      • /interfaces
        • _interface.scss
      • /plugins
        • _plugin.scss
    • _project.scss
    • styles.scss

The only files that should exist outside of a folder in your /styles root are _project.scss files that act as manifests for a given project's SCSS. The value of "project" should always match a child folder in /styles (i.e. for the "Shopping App", the project identifier may be "shop" which results in a file _shop.scss in the root folder as well as a folder named shop with any shopping-specific components, interfaces, and plugins defined in their relevant folders.

The base folder is a Git Subtree that is shared across all Choose Energy projects. If you have not worked with Git Subtrees before, you should read git subtree: a tutorial before attempting to modify our base styles.

It should be noted that no interface level styles should ever be defined within the base styles; if a specific interface is meant to be reusable across projects, it should be built as related components instead. Similarly, foundational styles should only be defined in our base styles. We should never be defining blanket overrides to foundational styles in projects. Any overrides to foundational styles should be scoped to the interface level.

Sizing

All sizing should be done using em values instead of px values to properly scale interfaces using zoom and accessibility features. Pixel values are acceptable for border values, border-radius values, and other em-fractional values that are not intended to scale without manual scaling.

Class Naming Patterns

Utility Classes

The basic syntax is as follows: x-<utilityName>.

The utilityName value must always be camelCase and prefixed with the x- namespace.

Utility classes modify low-level structural and positional traits of elements, and their usage is most frequent when prototyping or designing in the browser. Utility classes may be applied directly to any element, though they may not always have an effect (i.e. x-textLeft will apply text-align to any element, but an img element would be unaffected by this utility class).

Every Utility Class is paired with a SCSS mixin that may be included in any element's property declarations to apply the utility. Using the utility mixin is the preferred method, but these Utility Classes exist specifically to allow a designer or developer the option to prototype using the utility classes without spending an egregious amount of time experimenting with different SCSS mixins.

Utility Classes should be avoided for use in production unless the only styling an element will receive is via the utility class.

It should be noted that a lot of companies espouse the u- namespace for their utility classes, but we use the Microformats2 parsing specification for specifying semantic microformats, which would have conflicts with the u- namespace.

JavaScript Classes

The basic syntax is as follows: js-<targetName>.

The targetName value must always be camelCase and prefixed with the js- namespace.

JavaScript-specific classes reduce the risk that changing the structure or theme of components will inadvertently affect any required JavaScript behaviour and complex functionality. It is not necessary to use them in every case, just think of them as a tool in your utility belt. Variant Classes

The basic syntax is as follows: v-<{variant-name}>.

Used hand in hand with component classes to account for variants in the look of a component.

Try to limit the number of variants in use on any particular component. While there are some generic variants that can be stacked together (for example .v-bigtalk and .v-bg-light on a editorial block), if you find that you are adding multiple variants, it is best to create a custom variant specific to that instance to reduce the chance of error.

Currently our icon font uses a combination of Choose Energy specific icons and a handful of general icons from the free font librarys available through the app, primarily from Fontawesome and Entypo. Long term we will likely replace those icons with our own versions for a more consistent look and feel, but the class names and character mapping should remain unchanged.

Files

The custom fonts generated by Fontastic can be found in fonts/ce-icons.{{extensions}}. There are four font files and all of them need to be replaced when the font is updated.

The $font-family-icon variable is defined in styles/base/globals/_typography.scss and shouldn't be updated unless the font-family name is changed in the font-face definition.

The font-face definition, specific icon class names, and character mapping is in styles/base/globals/_icons.scss. The file needs to be edited with the new class names and character mapping every time a new icon is added to the font.

The generated icon classes all use the format c-icon-<icon-name>. The generic icon class overrides any font stylings (except size) that might be applied to the icon via other classes.

How to Update

Using Fontastic makes adding new icons to the font is relatively straightforward, but there are still a couple of important considations to note when doing so.

Creating the icon SVG

When new SVG icons are created to add to the font, they should be laid out on a square 100x100 px artboard and centered vertically and horizontally (in certain cases, we may instead want to vertically align the icons to a consistent baseline). The icon should be propotionally scaled so that the larger of the width or height is equal to 100px and therefore the icon takes up the maximum space on the artboard without being distored.

Our custom svg icons are stored in Google Drive under Marketing/Brand/Creative Assets/Choosicons. The file names for the svgs of each icon should follow the format <icon-name>.svg or <icon-name>-<icon-name>.svg. This name will be used to auto-generate the class name for the icon, so it should be descriptive of how the icon will be used on the site.

Fontastic Settings

Next, the file should be uploaded to the Choose Energy icon set in Fontastic. This option is found under the menu to the top right of the icon set. DO NOT delete the set!!

After a new icon(s) is added, they will appear in the set as unselected (with a gray background). To add it to the font, it must first be selected - if being used as a replacement for a Fontawesome or Entypo icon, take a note of the class name used for that icon so it can be assigned to the new icon instead.

Warning: Try not to unselect and reselect icons unless you're planning on making changes to that icon - this can cause unnessary changes to occur to the character mapping or class names, and potentially introduce errors. If you do remove and re-add an icon, double check these settings on the "Customize" tab.

Once you've selected all the icons you want to include in the font, move to the "Customize" tab to check the class and character mapping. The class names should be auto-generated from the file name, but they can be altered here if necessary.

Updating the files in ChooseUX

If everything looks right under the "Customize" tab, move on to the Publish tab and click the button to download the font and install it manually. You'll download a .zip file containing the 4 font files, a stylesheet, and a reference icon reference page.

First, copy the fonts to the fonts folder, replacing the current ce-icons font files.

Next, open up the styles.css file in the downloaded folder and copy the new class definitions from this file to styles/base/globals/_icons.scss. The classes for new icons will be at the end of the downloaded stylesheet, so if that's the only change, then you can just copy the new clases over. However, if you've done any re-mapping of the character codes, changed the class names, or changed the icons selected in Fontastic it will be safer to re-copy over all the class definitions.

In order to see the changes to the font file on the site, you'll need to re-run grunt server in the terminal. You'll also want to run grunt verify to see if you need to do any clean-up to the copied classes to conform with our css style guide (for example, you'll need to replace the double quotes in the copied class definitions to single quotes).

After this, you're free to begin using the icons on the site!

Grid System

We maintain our own grid system, as some of our requirements are unique compared to most of the framework-based grid systems. The primary goals of the Choose Grid System are as follows:

  • Lightweight: We don't need a grid system that immediately enables every potential grid layout that we might need. We know our needs and can optimize our grids for those needs.
  • Responsive: Nobody wants a layout that breaks when the browser gets resized. To combat this, we've predefined our breakpoints and set our grid system up with the ability to stack grid items when necessary.
  • Easy to use: Built to be easy to learn, easy to implement, ChooseGrid is very predictable with its defaults but can be configured for much more advanced usage.
  • Semantic: Many grid systems muck up the DOM with non-semantic class names that define layout styles in HTML, which is a practice we actively discourage. ChooseGrid helps keep separation of concerns by providing SCSS Mixins for every grid definition. When defining production-ready code, we can utilize the Mixins and ensure our DOM remains semantic and keeping class assignments to a minimum.
  • Flexible: While being lightweight, we still want the grid system to be flexible enough to produce a wide range of layouts without additional cost. To do this, we provide pre-defined layout classes to enable prototyping layout variants quickly. These pre-defined grid classes should be used only in prototyping and should never make it to production.
  • Cross-browser: Initially, we were experimenting with using flexbox grids for our apps, but testing proved out that the polyfills were still insufficient for providing a single experience across browsers we support. The current ChooseGrid implementation should function properly on all browsers we support.

Accessibility

For us to reach the widest possible audience for our products, we need our products to be accessible to all people, regardless of limitations on their abilities. Any user should be able to navigate the product, understand it, and complete their goals. Accessibility is hard, and this is only an overview.

Assistive technologies have enabled individuals with disabilities to use technology in new and innovative ways. When thinking about our products, we should always consider how individuals will be using them in the following ways:

  • Without sound
  • Without color
  • With the high contrast mode enabled
  • With the screen magnified
  • With a screen reader (no visible screen)
  • With voice control only
  • With a combination of the above

Here are some general rules for designing and building highly accessible products to which we should strive to adhere:

  • Design and build spartan interfaces: Our products are all very task-oriented. Every user interface should have the least number of elements required to complete the task effectively. Do not add decorative elements without considering and measuring their impact on the user's ability to complete the task or tasks that they are looking to complete.
  • Do not strand your user: Consider the "back" and "reload" buttons in a web browser to be the nuclear option to navigation. Avoid leaving your user stuck in an interface with no way back, getting thrown to a view out of context, or not being given proper time to consider an alert or confirmation screen. For example, if a user adds a product to their cart and is taken to checkout, the checkout screen should have a way back to the previous shopping screen, they should be taken to the top of the checkout page, and it should be clear to the user why they are now at a checkout page.
  • Support non-standard control schemes: Some users do not use mice and rely heavily on keyboard navigation, some users rely on voice recognition and screen readers, and some users will be heavily reliant on touch interfaces. Ensure that :hover and mouseover events are able to be triggered by these users just like those using a mouse. Ensure that any action available via a keyboard shortcut is also available in the user interface.
  • Ensure touch targets are usable: Touch targets should be a minimum of 48x48 device pixels, and there should be at least 8 device pixels between any two touch targets. In most cases within our products, this equates to 3em square or larger touch targets with 0.5em margins around them.
  • Use semantic markup: Don't rely only on visual hierarchy to convey the information architecture of a given component or page. Utilize highly semantic markup that ties related information together at the DOM level so screen readers and individuals with styles disabled are able to understand the information on the page.
  • Support screen magnification: Visually impaired users may increase the default font size on their device or zoom in on content using scalar magnification. Ensure that their experience is not hindered by missing elements, broken or overlapping interface components, and that text properly scales across different platforms.
  • Support color-blind interfaces: When an interface relies heavily on color for the interpretation of the interface, add iconography or patterns to accompany colors to allow users with color-blindness to properly interpret the interfaces.
